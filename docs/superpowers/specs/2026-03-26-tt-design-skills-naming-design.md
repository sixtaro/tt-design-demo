# tt-design Skills 命名与入口设计

## 背景
当前项目级技能位于 `.claude/skills/` 下，现有命名为 `tt-*`。随着技能数量增加，现有命名方式在品牌一致性、可发现性和后续扩展上不够统一，因此需要建立一套稳定的项目级技能命名规范，并引入一个轻量的技能总入口。

## 目标
- 将项目级技能统一命名为 `tt-design-xxx`
- 直接替换旧技能名，不保留兼容别名
- 新增一个仅负责导航分发的总入口技能
- 保持各技能职责清晰，避免总入口膨胀成重复文档

## 现状
当前 `.claude/skills/` 下已有以下技能：
- `tt-create-component`
- `tt-generate-docs`
- `tt-generate-storybook`
- `tt-manage-version`
- `tt-webapp-testing`

这些技能已经覆盖组件创建、文档生成、Storybook、版本管理和 Web 侧验证等场景，但命名仍混用 `tt-*` 前缀，尚未体现 `tt-design` 这一项目语义。

## 设计决策

### 1. 命名规范统一为 `tt-design-xxx`
将现有技能重命名为：
- `tt-design-create-component`
- `tt-design-generate-docs`
- `tt-design-generate-storybook`
- `tt-design-manage-version`
- `tt-design-webapp-testing`

#### 设计理由
- 与项目名 `tt-design` 对齐，更清楚地表达这些技能属于当前仓库
- 比 `tt-*` 更稳定，后续继续新增技能时不容易语义漂移
- 搜索和识别时更容易按统一前缀聚合

### 2. 旧名直接替换，不保留兼容别名
本次改动采用直接替换策略：旧技能目录名和 skill name 一并更新，不保留旧别名。

#### 设计理由
- 避免新旧命名长期并存，导致调用口径混乱
- 项目级技能数量较少，统一替换的维护成本可控
- 可以借这次调整一次性收口技能体系

### 3. 新增导航型总入口 `tt-design-routing`
新增一个轻量 skill：
- `tt-design-routing`

该 skill 只做导航和路由，不承载具体实现细节。

#### 职责范围
- 根据用户任务判断应调用哪个 `tt-design-xxx`
- 提供简洁的场景映射
- 明确要求继续加载对应专项 skill，而不是自己重复执行各专项流程
- 当一个请求同时涉及多个意图时，优先指向主技能，并在文案中提示可能需要继续调用的次级技能

#### 最低实现要求
`tt-design-routing` 的 `SKILL.md` 至少应包含：
- 一段极短概述，说明它是 tt-design 项目技能入口
- 一份“场景 → 目标 skill”映射表
- 明确的要求语句：命中场景后必须继续调用对应 `tt-design-xxx`
- 一段边界说明：它不替代 `CLAUDE.md`，也不替代各专项 skill

#### 非职责范围
- 不重复各专项 skill 的完整流程
- 不承载仓库长期规则
- 不替代 `CLAUDE.md`
- 不做项目百科式说明

#### 设计理由
- 用户或模型在面对项目任务时，先有一个统一入口更易发现
- 用 `routing` 明确表达“导航型 skill”，避免总入口语义过宽
- 降低后续新技能加入时的入口维护复杂度

## 技能边界划分
- `CLAUDE.md`：长期稳定的仓库规则、开发约束、代码规范
- `tt-design-routing`：技能选择与导航分发
- 各 `tt-design-xxx`：单一领域的专项流程与检查点

这种分层能保证：
- 仓库规则不被技能文档重复拷贝
- 总入口足够轻，不会变成第二份 CLAUDE.md
- 专项 skill 继续保持聚焦

## 推荐落地清单
实施时应完成以下事项：
1. 重命名技能目录
2. 更新每个 `SKILL.md` frontmatter 中的 `name`
3. 同步更新标题与正文中的技能名
4. 新增 `tt-design-routing/SKILL.md`
5. 搜索并替换仓库内对旧 skill 名的引用
6. 校验 skill 列表中是否已显示新名称

### 迁移范围
由于本次采用“旧名直接替换”，实施时至少要覆盖以下引用面：
- `.claude/skills/**/SKILL.md` 中互相引用的 skill 名
- `.claude` 目录下其他说明文件或辅助文档
- `docs/` 下与 skill 使用相关的说明、spec 或计划文档
- 其他显式写死旧技能名的提示词、命令说明或自动化文本（如存在）

### 验收标准
本次重命名完成后，应满足以下完成态：
- 不再存在仍在使用中的旧 `tt-*` 技能名引用
- 所有重命名后的技能都能在技能列表中被发现
- `tt-design-routing` 只指向新技能名，不再引用旧名称
- 技能目录名、frontmatter `name`、标题文案三者保持一致

## 风险与约束
- 旧名不保留兼容后，任何仍引用旧技能名的内容都必须同步修正
- 总入口若写得过重，会与各专项 skill 或 `CLAUDE.md` 重复，因此必须控制篇幅和职责边界
- `name` 需要继续遵循 skill 命名约束：只用字母、数字和连字符

## 最终结论
采用以下最终方案：
- 技能命名统一为 `tt-design-xxx`
- 旧名直接替换
- 新增轻量导航入口 `tt-design-routing`
- 总入口只负责路由，不承载详细规则

该方案在统一性、可发现性、维护成本和职责清晰度之间最平衡，适合作为当前仓库项目级技能体系的标准命名与入口结构。
